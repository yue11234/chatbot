│  └─────────────┬──────────────────┬────────────────────┘  │
│                │                  │                        │

│    └────────┬─────────┘  └────────────────────────────┘  │
│             │                                              │
│    ┌────────▼──────────────────────────────────────────┐  │

│    └────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────┘

          ┌───────────────┼───────────────┐
          ▼               ▼               ▼
    ┌──────────┐   ┌──────────────┐  ┌─────────┐

### 1. 入口与配置

#### `main.py` — FastAPI 应用工厂

**职责：** 创建 FastAPI 应用实例，注册路由，配置 CORS 跨域中间件。

**思路：** 采用工厂模式，统一管理中间件和路由注册，便于后续扩展。

**关键代码位置：** [backend/app/main.py](backend/app/main.py)

---

#### `run_server.py` — 服务启动入口

**职责：** 加载 `.env` 环境变量，处理 Windows 事件循环兼容性（`asyncio.WindowsSelectorEventLoopPolicy`），启动 Uvicorn 服务器。

**关键代码位置：** [backend/app/run_server.py](backend/app/run_server.py)

---

#### `core/config.py` — 全局配置管理

**职责：** 使用 `pydantic-settings` 从 `.env` 文件读取所有配置项，提供类型安全的配置访问。

**支持的配置项：**

| 配置键 | 说明 |
|---|---|
| `DATABASE_URL` | 数据库连接字符串（SQLite / MySQL）|
| `DEFAULT_MODEL` | 默认使用的大模型名称 |
| `OPENAI_API_KEY` | OpenAI API 密钥 |
| `DASHSCOPE_API_KEY` | 阿里云通义千问 API 密钥 |
| `DEEPSEEK_API_KEY` | DeepSeek API 密钥 |
| `EMBEDDING_MODEL` | Embedding 模型名称（默认 bge-m3）|
| `CHROMA_PATH` | ChromaDB 本地存储路径 |
| `OLLAMA_BASE_URL` | Ollama 服务地址 |

**关键代码位置：** [backend/app/core/config.py](backend/app/core/config.py)

### 2. API 层

#### `api/chat_routes.py` — 聊天接口路由

**职责：** 暴露两个核心聊天接口：

| 接口 | 方法 | 说明 |
|---|---|---|
| `/chat/invoke` | POST | 同步调用，等待完整响应后返回 |
| `/chat/stream` | POST | 异步 SSE 流式输出，逐 Token 推送到前端 |

**思路：**
1. 接收前端传入的 `message`、`thread_id`、`agent_id` 等参数
2. 通过 `get_agent(agent_id)` 获取对应的 LangGraph 智能体实例
3. 使用 `thread_id` 实现多轮对话连续性（每个对话线程独立维护上下文）
4. 支持 LangGraph 中断恢复（`interrupt`）机制
5. 流式接口通过 `astream_events` 实时推送 Token 事件

**关键代码位置：** [backend/app/api/chat_routes.py](backend/app/api/chat_routes.py)

---

#### `api/schema/chatSchema.py` — 数据结构定义

**职责：** 使用 Pydantic 定义请求和响应的数据模型。

| 模型 | 说明 |
|---|---|
| `UserInput` | 用户输入（message、thread_id、agent_id、agent_config）|
| `StreamInput` | 流式输入，继承 UserInput，增加 `stream_tokens` 字段 |
| `ChatMessage` | 响应消息（type、content、tool_calls、run_id）|
| `ToolCall` | 工具调用信息（name、args、id）|

**关键代码位置：** [backend/app/api/schema/chatSchema.py](backend/app/api/schema/chatSchema.py)

### 3. AI 核心层

#### 3.1 LLM 工厂

##### `ai/llm.py` — 大模型工厂函数

**职责：** 根据模型名称，返回对应的 LangChain LLM 实例，屏蔽多个提供商的差异。

**支持的模型提供商：**

| 提供商 | 模型示例 | LangChain 集成包 |
|---|---|---|
| OpenAI | GPT-4o、GPT-4o-mini | `langchain-openai` |
| DeepSeek | deepseek-chat | `langchain-deepseek` |
| 阿里通义千问 | qwen-plus、qwen-max | `langchain-community`（DashScope）|
| Ollama 本地 | 任意本地模型 | `langchain-ollama` |
| Fake（测试用）| — | LangChain 内置 |

**思路：** 工厂函数 `get_model(model_name)` 根据传入的 `ModelName` 枚举值分支返回对应实例，所有模型均启用流式输出（`streaming=True`）。

**关键代码位置：** [backend/app/ai/llm.py](backend/app/ai/llm.py)

---

##### `ai/models.py` — 模型名称枚举

**职责：** 用 `StrEnum` 定义各模型提供商下的可用模型名称，供工厂函数和配置读取使用。

**关键代码位置：** [backend/app/ai/models.py](backend/app/ai/models.py)

---

#### 3.2 智能体模块

##### `ai/agent/agents.py` — 智能体注册表

**职责：** 维护一个字典形式的智能体注册表 `DEFAULT_AGENTS`，提供 `get_agent(agent_id)` 和 `get_all_agent_info()` 两个核心函数。

**思路：** 采用注册表模式，新增智能体只需在字典中添加一条记录，API 层无需修改。前端通过 `get_all_agent_info()` 动态获取可用智能体列表。

**关键代码位置：** [backend/app/ai/agent/agents.py](backend/app/ai/agent/agents.py)

---

##### `ai/agent/paper_assistant.py` — 论文助手智能体

**职责：** 一个支持 RAG 知识库检索的智能体，用于回答论文领域相关问题（图像翻译、扩散模型等）。

**框架：** LangGraph `StateGraph`

**工作流：**

```
用户输入
    │
    ▼
model 节点（调用 LLM，携带 System Prompt）
    │
    ├── 有 tool_calls → tools 节点（执行 search_papers 工具）→ 回到 model 节点
    │
    └── 无 tool_calls → 结束，输出最终回复
```

**关键设计：**
- System Prompt 指示模型优先使用知识库工具搜索，再组织回答
- 使用 `MemorySaver` 作为 Checkpointer，支持基于 `thread_id` 的多轮对话记忆
- 工具绑定：`model.bind_tools([search_papers])`

**关键代码位置：** [backend/app/ai/agent/paper_assistant.py](backend/app/ai/agent/paper_assistant.py)

---

##### `ai/agent/multi_agent.py` — 多智能体协作

**职责：** 基于 LangGraph Supervisor 模式，协调多个专属子智能体共同完成任务。

**框架：** LangGraph + `langgraph-supervisor`

**架构：**

```
用户问题
    │
    ▼
Supervisor（路由决策）
    │
    ├── 数学问题 → math_agent（数学计算与推理）
    ├── 代码问题 → code_agent（代码生成与调试）
    └── 通用问题 → general_agent（通用知识）
```

**关键设计：**
- 通过 `create_supervisor()` 自动生成 Supervisor 路由逻辑
- 三个子智能体各自有独立的 System Prompt 限定职责范围
- 使用 `InMemorySaver` 维护对话状态
- 配置 `parallel_tool_calls=False` 保证工具调用顺序执行

**关键代码位置：** [backend/app/ai/agent/multi_agent.py](backend/app/ai/agent/multi_agent.py)

---

##### `ai/agent/oa_assistant.py` — OA 助手智能体（扩展示例）

**职责：** 演示如何结合结构化数据库（员工信息）与非结构化知识库（员工手册 PDF）构建企业 OA 助手。

**关键代码位置：** [backend/app/ai/agent/oa_assistant.py](backend/app/ai/agent/oa_assistant.py)

---

#### 3.3 RAG 模块

##### `ai/rag/chromaClient.py` — 向量数据库客户端

**职责：** 初始化 ChromaDB 持久化客户端，创建/加载向量集合，绑定 Embedding 模型。

**思路：** 使用 Ollama 本地部署的 `bge-m3`（或 `mxbai-embed-large`）生成文本向量，写入 ChromaDB 的 `handbook` 集合。`bge-m3` 支持中英双语，适合中文文档处理。

**关键代码位置：** [backend/app/ai/rag/chromaClient.py](backend/app/ai/rag/chromaClient.py)

---

##### `ai/rag/hybridRetriever.py` — 混合检索器

**职责：** 结合 BM25（关键词）与向量（语义）两种检索方式，提升知识库召回质量。

**思路：**
- **BM25**：擅长精确关键词匹配，适合专有名词、术语查询
- **向量检索**：擅长语义理解，适合模糊问题和近义词查询
- 通过 LangChain `EnsembleRetriever` 融合两路结果，默认权重：BM25 40%，向量 60%
- 使用单例模式 + `asyncio.Lock` 实现懒加载，首次调用时从 ChromaDB 加载文档并构建 BM25 索引

**关键代码位置：** [backend/app/ai/rag/hybridRetriever.py](backend/app/ai/rag/hybridRetriever.py)

---

##### `ai/rag/ingestPdfs.py` — PDF 文档写入管道

**职责：** 读取 PDF 文件，切分文本，生成向量，写入 ChromaDB。

**思路：** PDF → 文本切片（Chunk）→ Embedding → ChromaDB 持久化。是知识库构建的数据准备脚本，使用前需先执行一次。

**关键代码位置：** [backend/app/ai/rag/ingestPdfs.py](backend/app/ai/rag/ingestPdfs.py)

---

#### 3.4 工具模块

##### `ai/tools/paper_tools.py` — 论文检索工具

**职责：** 封装为 LangChain `@tool`，供 LangGraph 智能体调用，从知识库检索相关内容。

**思路：** 调用混合检索器 `get_hybrid_retriever()` 查询 ChromaDB，格式化返回检索结果（包含来源文件、页码、内容摘要）。工具描述（docstring）直接影响 LLM 决策是否调用该工具。

**关键代码位置：** [backend/app/ai/tools/paper_tools.py](backend/app/ai/tools/paper_tools.py)

### 3. 自定义 Hooks

#### `chat/hooks/useStreamChat.ts` — SSE 流式接收

**职责：** 封装与后端 `/chat/stream` 接口的 SSE 通信逻辑。

**思路：**
1. 向 `/chat/stream` 发送 POST 请求，接收 `ReadableStream`
2. 逐行解析 SSE 事件（`data: {...}` 格式）
3. 处理不同事件类型：
   - `token`：追加到当前消息末尾（流式显示）
   - `message`：写入完整消息（含 Tool Call 信息）
   - `error`：展示错误提示
   - `end`：标记流式输出结束
4. 组件卸载时自动取消请求（`AbortController`）

**关键代码位置：** [frontend/app/chat/hooks/useStreamChat.ts](frontend/app/chat/hooks/useStreamChat.ts)

---

#### `chat/hooks/useChatActions.ts` — 聊天行为处理

**职责：** 封装发送消息的业务逻辑，包括构建请求体、调用流式 Hook、更新消息列表。

**关键代码位置：** [frontend/app/chat/hooks/useChatActions.ts](frontend/app/chat/hooks/useChatActions.ts)

---

#### `chat/hooks/useScrollToBottom.ts` — 自动滚动

**职责：** 监听消息列表变化，自动将聊天窗口滚动到最新消息位置。

**关键代码位置：** [frontend/app/chat/hooks/useScrollToBottom.ts](frontend/app/chat/hooks/useScrollToBottom.ts)

## 数据流与通信

### 完整消息流转路径

```
1. 用户输入文字，点击发送
        │
        ▼
2. useChatActions 构建请求体
   { message, thread_id, agent_id, stream_tokens: true }
        │
        ▼
3. useStreamChat 发起 POST /chat/stream（SSE 连接）
        │
        ▼
4. FastAPI chat_routes.stream() 接收请求
        │
        ▼
5. _handle_input() 解析参数
   - 获取/创建 thread_id（多轮对话标识）
   - 调用 get_agent(agent_id) 获取对应智能体
        │
        ▼
6. agent.astream_events() 执行智能体
        │
        ├── paper-assistant 流程：
        │   a. LLM 接收消息 + System Prompt
        │   b. 决定是否调用 search_papers 工具
        │   c. 工具调用 → hybridRetriever 查询 ChromaDB
        │   d. 将检索结果注入 LLM 上下文重新生成回答
        │   e. 逐 Token 推送到 SSE 流
        │
        └── multi-agent 流程：
            a. Supervisor 分析问题类型
            b. 路由到 math/code/general 子智能体
            c. 子智能体生成回答
            d. 逐 Token 推送到 SSE 流
        │
        ▼
7. message_generator() 格式化 SSE 事件推送
   event: token  data: {"content": "..."} （流式 Token）
   event: message data: {"type": "ai", ...}（完整消息）
   event: end    data: {}
        │
        ▼
8. 前端 useStreamChat 解析 SSE 事件
   - token → 追加到消息末尾（实时显示）
   - message → 写入完整消息（含 Tool Call）
   - end → 标记流式结束
        │
        ▼
9. MessageBubble 渲染 Markdown + 代码高亮
10. 消息持久化到 localStorage（按 thread_id 分区）
```

## 技术选型说明

| 技术 | 选型理由 |
|---|---|
| **LangGraph** | 相比 LangChain Expression Language（LCEL），LangGraph 以有向图形式建模智能体状态机，天然支持循环调用、中断恢复、多智能体协作；适合复杂 Agent 流程编排 |
| **FastAPI** | 原生异步支持（async/await），高性能 ASGI 框架；内置 Pydantic 数据验证；SSE 流式输出友好；自动生成 OpenAPI 文档 |
| **ChromaDB** | 轻量级本地向量数据库，零配置启动，持久化到本地文件；适合中小规模知识库；与 LangChain 深度集成 |
| **BM25 混合检索** | 纯向量检索在专有名词上召回不稳定；BM25 + 向量混合能兼顾关键词匹配和语义理解，提升知识库问答质量 |
| **bge-m3 Embedding** | 支持中英双语，向量维度 1024，适合中文文档检索场景；通过 Ollama 本地部署，无需外部 API |
| **Next.js 14 App Router** | 支持 SSR/CSR 混合，文件系统路由简洁；与 Ant Design 5 集成后 FOUC 问题已有官方解决方案 |
| **SSE（Server-Sent Events）** | 相比 WebSocket 更轻量，适合单向推送（服务端 → 客户端）；HTTP 友好，不需要额外握手协议；适合流式 LLM 输出场景 |
| **Pydantic Settings** | 类型安全的配置管理，支持从 `.env` 文件自动加载，字段验证明确，避免配置项写错不报错 |
| **SQLModel** | 同时继承 SQLAlchemy（ORM）和 Pydantic（验证），减少模型重复定义；异步支持良好 |
